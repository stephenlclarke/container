# fix(health): align probe scheduling and status reporting

## Type of Change

- [x] Bug fix
- [ ] New feature
- [ ] Breaking change
- [x] Documentation update

## Motivation and Context

This change implements the Compose-enabling runtime portion of [apple/container#1918](https://github.com/apple/container/issues/1918). The health observer already executes configured probes and exposes `ContainerSnapshot.health`, but its cadence and start-period behavior differ from Docker, and the CLI drops health while converting snapshots to `ManagedContainer`.

Accurate health state is required by Docker Compose `depends_on.condition: service_healthy`, `up --wait`, `start --wait`, `ps`, and health transition events. Running state alone is not a readiness signal for databases, brokers, and other services that need warm-up time.

## Implementation Details

- Delays the first probe by the active normal or start interval.
- Adds Docker's five-second default start interval.
- Uses the start interval only while the container remains `starting` inside the start period.
- Counts failures after a successful probe even when the original start period has not elapsed.
- Treats zero interval, timeout, and retry values as Docker defaults.
- Emits `health_status: healthy` and `health_status: unhealthy` only on state transitions.
- Adds current health to event attributes.
- Preserves health in `ManagedContainer`, JSON inspect output, and the `container list` table.

Compose orchestration policy remains outside `apple/container`; this change only provides generic container health behavior and queryable runtime state.

## Compatibility Notes

The change is additive on the wire. Containers without a health check continue to omit health. Existing `ContainerSnapshot.health` consumers keep the same enum shape, and legacy `ManagedContainer` JSON without a health field still decodes with `nil`.

Probe cadence becomes Docker-compatible: configurations that previously relied on an immediate first probe now wait for the applicable interval. This is the intended compatibility correction.

## Docker Compose Parity

This runtime state enables Compose to wait for health rather than liveness and to render current health through stable CLI JSON discovery. The companion Compose change adds explicit `up/start --wait` health regression coverage, marks the options supported, and adds live parity validation.

## Remaining Risks

- Probe output history and failing-streak details are not yet exposed as a public snapshot object.
- An on-demand `container healthcheck run` command remains future work under apple/container#1918.
- The API server continues to own health scheduling; Apple maintainers may prefer a lower runtime owner in a future design.
